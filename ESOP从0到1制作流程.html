📘 ESOP 从 0 到 1 制作流程

一份操作手册是怎么做出来的 — 实施顾问/培训用

美诗儿（浙江）环境智能电器有限公司 · MES 项目

① 准备阶段 ② 制作阶段 ③ 发布阶段 ⚠️ 注意事项
📌 核心原则
若有客制需求，以客制需求为准；没有客制需求的，以标准功能为主。
① 准备阶段 — 先做什么
动手制作之前，先把这些准备工作做好，后面才会顺利。
1
跟客户确认：做哪些模块？
和客户开会，确定需要制作操作手册的车间/部门范围。
以美诗儿为例，最终确定 5 大模块：注塑车间 组装车间 设备维保 模具维保 质量检验
💡 先做核心模块，后续可以再补充，不要一次贪多。
↓
2
收集材料
向客户收集以下材料：
📄 操作说明书 — Word 文档、PDF 简报（鼎捷标准功能说明书）
🖼️ 系统截图 — 操作界面的实际截图
📊 流程文件 — 流程图、操作流程说明
💡 材料越全，制作越省事。特别是截图，少了后面要补很麻烦。
↓
3
确认参照标准
判断标准：
✅ 如果客户有客制需求（定制开发的功能）→ 以客制需求确认书的内容为准
✅ 如果客户用的是标准功能 → 以鼎捷标准操作说明书（PL 文档）为准
⚠️ 两者混合时，客制优先，标准功能做补充
↓
4
准备工具
让开发人员帮忙搭建好环境：
🛠️ Python — 运行生成器脚本
🛠️ VSCode — 编辑内容
🛠️ Git — 版本管理
🛠️ GitHub 账号 — 代码托管
🛠️ Cloudflare 账号 — 部署上线
② 制作阶段 — 怎么做
这是核心阶段，把准备好的材料变成操作手册。
5
梳理操作流程
把每个模块的操作流程拆解成标准结构：
📌 场景说明 — 这个模块是干什么的
📌 流程图 — 操作顺序一目了然
📌 操作步骤 — 按 1.1 → 1.2 → 1.3 一步步写清楚
📌 注意事项 — 容易出错的地方单独列出来
💡 步骤顺序要按现场操作人员的操作顺序来写，不是按系统菜单顺序。
↓
6
整理截图图片
从 PDF/Word 中提取截图：
① 从参考文档中提取操作界面的截图
② 按步骤顺序统一编号：fig-01.png、fig-02.png...
③ 放入对应模块的 images/ 文件夹

关键：图片必须对上操作步骤的文字描述。
比如步骤写的是"进入工作平台"，配图就应该是工作平台的界面截图。
↓
7
编写步骤描述
用通俗易懂的语言写操作说明：
✅ 作业说明 — 这一步要做什么、怎么操作
✅ 备注 — 有什么要注意的、什么情况下会触发什么
✅ 配图 — 每步对应的截图

💡 写完后读一遍，想象自己是操作工，能不能看懂。
↓
8
生成 HTML 页面
内容写好后，运行一行命令：
python generate.py
脚本会自动：
✅ 生成每个模块的 HTML 页面
✅ 自动绘制流程图（横向，步骤→子步骤）
✅ 生成首页（卡片导航）
✅ 所有页面自动串联侧边栏导航
↓
9
检查效果
打开浏览器预览生成的页面，逐一核对：
✅ 图片加载是否正确
✅ 图片和文字描述是否对应
✅ 流程图步骤顺序、箭头方向是否正确
✅ 侧边栏导航是否完整
✅ 内容有没有错别字
↓
10
给客户确认
本地确认没问题后，发给客户预览：
① 启动本地服务器：python -m http.server 8000
② 客户通过局域网访问：http://你的IP:8000
③ 客户提出修改意见 → 调整内容 → 重新生成 → 再次确认

💡 预留 2~3 轮修改，客户通常会调整一些细节。
③ 发布阶段 — 让客户看到
确认无误后，部署到线上，客户通过网址就能直接访问。
11
提交到 Git 版本管理
git add -A
git commit -m "版本说明，比如：新增质量模块检验单指派"
git push
↓
12
部署到 Cloudflare Pages
发布到互联网，客户通过网址就能访问：
🌐 自动部署 — 推送到 GitHub 后，Cloudflare 自动部署
🌐 获得网址 — 如 mayshel-esop.pages.dev
🌐 全球加速 — 客户在任何地方打开都很快
🌐 手机/电脑/平板 — 都能正常查看
↓
13
后续更新
操作手册不是一次做完就完了，后续随时可以更新：
🔄 客户需求变了 → 修改内容
🔄 运行 python generate.py → 重新生成
🔄 git push → 自动部署 → 客户马上看到最新版
⚠️ 注意事项 — 容易踩的坑
⚠️ 1. 图片必须对上文字
这是最常见也是最重要的返工原因。
步骤写的是"进入工作平台"，配图就应该是工作平台界面，不能放别的。
客户可能不会逐字读你的文字，但一眼就能看出图片不对。
匹配方法：PDF 中某页文字描述什么操作 → 该页的截图就是对应步骤的配图。
⚠️ 2. 操作步骤要有闭环
每个操作流程要有头有尾，不能只写一半。
比如设备维修：加工中 → 故障 → 维修 → 加工中
又比如模具叫修：异常发现 → 叫修 → 签收 → 维修 → 完工 → 恢复可用
写完后检查：这个流程是不是完整的？有没有漏掉某个环节？
⚠️ 3. 用操作工能听懂的话
操作手册是给现场操作人员看的，不是给 IT 人员看的。
❌ "系统根据 QCFROMTYPE 和 INSPTYPE 进行数据校验判断"
✅ "选择检验单类型（PQC首检/巡检），系统自动匹配对应检验员"
写完后找不懂系统的人读一遍，卡壳的地方就是需要改的地方。
⚠️ 4. 不要手动改 HTML
永远不要直接修改 .html 文件。
所有内容写在 generate.py 的字典里，运行脚本自动生成。
手动改 HTML 的话，下次重新生成时手动改的内容会丢失。
改内容 → 运行脚本 → 重新生成 → 推送部署，保持这个习惯。
⚠️ 5. 每次修改记版更记录
每次修改了什么内容，写在 CHANGELOG.md 里。
好处：客户问"改了啥"可以直接看；出问题可以回退到上一个版本。
格式：日期 + 修改内容 + 修改人。
⚠️ 6. 客制 vs 标准
牢记原则：
✅ 有客制需求 → 以客制需求确认书为准
✅ 无客制需求 → 以标准功能操作说明书为准
✅ 两者混合 → 客制优先，标准做补充
客制需求通常记录在《需求确认书》中，标准功能参考鼎捷 PL 文档。
✅ 总结：一句话记住
📋

先准备

确认范围 → 收集材料 → 明确标准

✏️

再制作

梳理步骤 → 整理图片 → 编写描述 → 生成页面

🔁

再确认

自己检查 → 客户确认 → 修改迭代

🚀

最后发布

Git 提交 → 推送到 GitHub → Cloudflare 部署上线
