1

需求分析 — 确定模块与内容范围

输入：客户需求
1.1确定操作手册涵盖的车间/部门
与客户（美诗儿）确认需要制作操作手册的模块范围：
最终确定 5 大模块：🏭 注塑车间 · 🏭 组装车间 · ⚙️ 设备维保 · 🔧 模具维保 · 📋 质量检验
已剔除：IIoT 设备看板、系统基础配置（客户确认不需要）
输出：模块清单 决定：5 大模块
↓
1.2收集参考文档与素材
从客户处获取参考材料：
📄 Word 文档 — 各模块的原始操作说明文档（.doc格式）
📄 PDF 参考 — 鼎捷标准功能操作简报（PL102 设备稼动、PL205 模治具寿命管理）
🖼️ 截图素材 — 已有操作截图图片
源文件：*.doc PL102.pdf PL205.pdf
2

内容采集 — 从文档提取操作步骤

输入：Word / PDF
2.1Word 文档解析
使用 python-docx 库读取 Word 文档，提取章节标题、操作步骤描述文字。 遇到加密文档时使用 olefile + antiword 等方式进行解密读取。
工具：python-docx 工具：antiword
↓
2.2PDF 参考文档解读
对鼎捷标准 PDF 简报进行文字提取和页面分析：
🔍 使用 pdfminer.six + PyMuPDF (fitz) 提取 PDF 文字内容
🔍 逐页分析文字描述，找到与操作步骤匹配的内容
🔍 特别注意「设备稼动闭环」、「维修闭环」、「保养闭环」等核心业务流程
关键：文字-图片对应 工具：PyMuPDF
3

图片处理 — 提取并匹配截图

输入：PDF截图
3.1从 PDF 中提取截图
使用 PyMuPDF 的 page.get_images() 提取 PDF 中嵌入的所有图片。 筛选出有效截图（排除装饰性元素），按页面编号命名保存。
PL102 提取 80+ 张 PL205 提取 60+ 张
↓
3.2图片匹配（关键步骤）
将 PDF 中的文字描述与 HTML 中的操作步骤逐条对照匹配，确定每张截图对应的操作内容。
匹配方法：
① 找到 PDF 中描述某操作步骤的页面
② 提取该页上的主要截图
③ 验证截图内容是否与文字描述一致
示例：PDF P59 描述"在作业人员工作平台找到设备" → 对应 HTML 步骤 1.1 "进入工作平台"
最重要步骤 确保图文一致
↓
3.3图片转换与命名
将 JPEG 格式的 PDF 截图转换为 PNG 格式（PIL.Image.save()）， 按 fig-01.png、fig-02.png 规则统一命名， 放入对应模块的 images/ 子目录。
images/equipment/fig-01~16.png images/mold/fig-01~07.png
4

代码开发 — Python 生成器

核心：generate.py
4.1编写 HTML 生成引擎
开发 generate.py，实现以下核心功能：
📌 内容字典（CONTENT）— 用 Python dict 定义所有模块的结构化数据：标题、部门、图标、图片目录、场景说明、步骤列表、注意事项
📌 HTML 模板引擎— 使用字符串模板 + str.replace() 渲染页面
📌 SVG 流程图生成— 根据步骤数据自动绘制水平方向流程图，包含主步骤→子步骤箭头连线
📌 图文配对布局— 每步操作对应截图，图片在左/文字在右
generate.py ~580 行代码
↓
4.2CSS 样式设计
📌 布局：固定侧边栏(260px) + 主内容区全宽
📌 图片：max-height:500px，支持双图并排
📌 交互：鼠标悬停阴影、图片点击放大（Zoom Overlay）
📌 颜色：蓝色主色调(#1a73e8) + 浅灰背景
css/style.css 响应式设计
↓
4.3JavaScript 交互
📌 图片放大：点击图片打开全屏遮罩预览（支持 SVG 和 PNG）
📌 滚动监听：页面滚动时自动高亮当前章节
js/main.js
5

内容编写 — 结构化数据录入

核心：CONTENT 字典
💡 数据结构设计
每个模块都是一个字典，包含：title / dept / icon / img_dir / scenario(场景说明列表) / steps(主步骤列表，每个包含 title + sub_steps) / notes(注意事项)
5.1注塑车间 & 组装车间
流程：派工作业 → 上工作业 → 产线点检 → 报工作业
共 4 个主步骤，13+ 个子步骤，详细描述每个操作环节
↓
5.2设备维保（从 PL102 PDF 重构）
核心概念：设备稼动 — 状态变更与闭环管理
重新梳理为 3 个主步骤：
① 设备稼动（基础操作）→ ② 状态变更场景（设置/维修/保养/暂停闭环）→ ③ 设备保养作业
闭环原则：原状态 → 目标状态 → 回归原状态
维修闭环：加工中→故障→维修→加工中
重点重构 引用：PL102
↓
5.3模具维保（从 PL205 PDF 重构）
核心概念：模具全生命周期管理
重新梳理为 4 个主步骤：
① 异常叫修 → ② 维修执行（维修闭环）→ ③ 定期保养（保养闭环）→ ④ 报表查看
维修闭环：异常发现→叫修→签收→维修作业→完工确认→恢复可用
重点重构 引用：PL205
↓
5.4质量检验
流程：PQC 首件检验 → 过程检验 → 质量异常处置
2 个主步骤，11 个子步骤
6

生成与校验 — 输出 HTML

输出：HTML 页面
6.1运行生成器
执行 python generate.py，程序自动：
① 遍历 CONTENT 字典中的所有模块
② 对每个模块：渲染场景说明→生成 SVG 流程图→渲染步骤说明→渲染注意事项
③ 输出 HTML 文件到 pages/ 目录
④ 生成首页 index.html（模块卡片导航）
python generate.py 1 次运行生成 6 个页面
↓
6.2校验与修正
✅ 打开 HTML 页面检查排版是否正确
✅ 验证图片是否正常加载、图文是否对应
✅ 检查流程图箭头方向是否正确
✅ 验证侧边栏导航是否完整
✅ 客户反馈修正（如"图片不对"→重新匹配）
迭代循环 多次修正
7

部署上线 — 让客户访问

输出：可访问的网址
📌 最终方案选择
我们准备了 3 种部署方式，最终选择了 GitHub Pages + Cloudflare Pages 作为正式发布方案， 本地 HTTP 服务器作为临时查看方案。

📁 方案A：本地服务器

局域网内临时查看，电脑不能关机

python server.py

适合：开发调试 / 临时展示

🐙 方案B：GitHub Pages

代码推送到 GitHub 仓库，自动部署

git push origin main

适合：国内客户访问
网址：wanghongyec-beep.github.io/mayshel-esop

☁️ 方案C：Cloudflare Pages

连接 GitHub 仓库，自动 CDN 加速

连接 Git → 自动部署

适合：全球客户访问
网址：xxxx.pages.dev

7.1Git 版本管理
使用 Git 进行版本控制：
📌 初始化新仓库 → 添加远程地址 → 提交代码 → 推送到 GitHub
📌 SSH 密钥认证：生成密钥对 → 添加到 GitHub Settings → 免密推送
git init ssh-keygen git push
↓
7.2最终交付
🎯 客户通过网址直接访问操作手册，无需安装任何软件
🎯 所有页面在任何浏览器/手机/平板上均可正常打开
🎯 后续更新：修改 generate.py 中的内容 → 重新生成 → 推送即可
